Designing a Unified Frontend for Nautilus Trader + ML Extension
Architecture Overview and Goals
Nautilus Trader is a high-performance algorithmic trading engine (in Python/Rust) which intentionally does not include a GUI or visual interface in its core design
medium.com
. Our goal is to build a unified web-based UI that brings together Nautilus Trader’s live trading/strategy controls and the custom ML “brain” extension into one coherent system. This UI should treat the trading engine and ML extension as one integrated platform, not as two separate tools, so that a developer or trader can monitor and control everything in one place. Key objectives include: real-time visibility into market data and trades, insight into ML model predictions and confidence, the ability to adjust parameters or trigger actions (like starting a strategy or retraining a model), and traceability (linking strategy decisions to the ML model inputs that influenced them). From a high level, the system will be deployed in Docker containers – one (or more) for Nautilus Trader and one for the ML extension (plus possibly others for services like databases or monitoring). The UI will run as a web app (e.g. a React frontend served by a lightweight web server container). The challenge is designing how these components communicate and how the UI is structured to reflect the combined “Nautilus+ML” functionality. We need to incorporate patterns for live data streaming, metrics collection, and user interactions across both domains (trading and ML). In essence, the UI will function as a control panel and dashboard for a “trading engine with an AI co-pilot.”
Integrated System Architecture
To bridge the Nautilus and ML components, we recommend a unified backend API layer (or API gateway) that the frontend interacts with. This layer can aggregate data and commands across the containers. For instance, the Nautilus container might expose REST endpoints or a WebSocket for trading data and control (e.g. start/stop strategies, fetch positions), and the ML container might expose its own API for model predictions or to trigger training. The API gateway (which could itself be a small service or even part of the UI server) can present a single cohesive API to the frontend. This hides the complexity of multiple services and ensures the frontend doesn’t need to manage multiple connections. It also simplifies implementing cross-cutting features like authentication or combined data responses (e.g. a single dashboard summary endpoint). Proposed high-level architecture for the unified Nautilus Trader + ML system UI. The Web UI interacts via a unified API layer (or gateway) with both the Nautilus trading engine and the ML extension. Real-time events flow to the UI via WebSockets (for immediate updates), while a metrics system (Prometheus) collects time-series data from both components for dashboard visualizations and alerts. Live data handling: For real-time streaming of events (trades, price ticks, model signals), the UI will use WebSockets (or Server-Sent Events) offered by the backend. WebSockets enable the UI to receive updates the moment something happens (e.g. a new order executes or a fresh ML signal arrives) without constant polling. This is crucial for a trading dashboard – “real-time dashboards aren’t just flashy — they’re critical for monitoring, trading, analytics, or any system where data changes in seconds”
medium.com
. Nautilus Trader can be extended to push key events to a WebSocket server (for example, via an internal message bus publishing to an external socket). Similarly, the ML extension can stream its live predictions or alerts (e.g. “model confidence dropped below threshold”) to the UI. By maintaining persistent WebSocket connections, the UI can display tick-by-tick updates like price changes, order book events, trade confirmations, and model outputs with minimal latency. Metrics and historical data: In addition to event streams, we will employ a time-series metrics database (like Prometheus) to collect aggregated data from both Nautilus and the ML extension. Each component will expose instrumented metrics (e.g. current P&L, latency of trade execution, number of trades, ML model inference latency, rolling accuracy, etc.) on an HTTP endpoint. Prometheus (running as another container) will poll these endpoints and store the metrics. Prometheus is a proven open-source monitoring toolkit that collects metrics from configured targets at intervals and can evaluate queries or trigger alerts
geekpaul.medium.com
. It essentially acts as a time-series datastore for numeric data, well-suited to track things like performance over time, strategy drawdown, or model drift. (Internally, Prometheus is like a time-series DB; such databases are optimized for timestamped data like server metrics and can even record events like trades over time
geekpaul.medium.com
.) The UI can either query Prometheus directly (using its HTTP API) or embed Grafana panels for rich visualization. Grafana is commonly paired with Prometheus to create real-time dashboards of system metrics. In our UI, we might not require a full Grafana installation; instead, we can integrate charts that query Prometheus for data. For example, a P&L chart of each strategy, or a time-series of an ML model’s prediction confidence vs actual price, can be rendered by fetching metric data from Prometheus. This approach decouples heavy data storage from the UI – the UI just visualizes data, while Prometheus handles collection and retention. Communication between Nautilus and ML: The architecture should also allow Nautilus and the ML extension to talk to each other in real-time. This could be done via an internal message bus or direct API calls. For instance, Nautilus could publish market data and technical indicator values onto a Redis or NATS message queue that the ML service subscribes to. The ML service processes these (possibly aggregating features) and emits back signals or decisions (e.g. a predicted price trend or a recommended position). Nautilus strategies can subscribe to those signals (either via the same message bus or by polling the ML service’s API). This decoupled but synchronous communication ensures the trading engine stays responsive. (Notably, Freqtrade’s design is similar: the trading engine supplies data to the ML module and gets predictions asynchronously so as not to block critical trading loops
medium.com
medium.com
.) In our architecture, this message flow means the UI can also tap into the bus for logging or debugging – for example, showing the raw ML signal values alongside trades.
Frontend Component Layout and Views
The UI will be organized into multiple views (pages) or at least distinct dashboard sections, each addressing a facet of the system. Navigation can be via a sidebar or tabs, allowing the user to switch between an overview and detailed views for trading or ML. We will use clear, logical sections so users can quickly find information, similar to how other platforms structure complex information (for example, an Airflow UI separates DAG views, and Grafana uses multiple dashboards). Below is a recommended breakdown of the UI components:
Main Dashboard (Unified Overview): This is the landing page giving a high-level summary of the entire system’s state. It would display key real-time KPIs and statuses from both Nautilus and the ML extension. For example:
Running Strategies Summary: A list of active trading strategies/algorithms, each with its current status (running/paused), current P&L, number of open positions, etc. This provides a quick health check of Nautilus Trader.
Market Snapshot: Key market data at a glance (especially if the system trades multiple instruments). Could be a ticker tape or small quote panels for the main markets being traded (price, volume, etc.), indicating that data ingestion (Databento feeds, etc.) is working.
ML Model Status: An overview of the ML “brain” – e.g. which model is currently deployed or active, its latest prediction or signal for the primary asset, and an indicator of its confidence or uncertainty. For instance, a gauge or dial showing “Bullish vs Bearish” signal strength, including uncertainty.
Alerts/Notifications: Any important alerts from either component, such as “Broker connection lost” (from Nautilus) or “Model drift detected, retraining recommended” (from ML). These could be shown as notifications or highlighted banners.
System Health Metrics: High-level system metrics like overall latency, message queue backlog, or resource usage if relevant – possibly using small sparkline charts. (These metrics would come from Prometheus – e.g. a chart of messages processed per second.)
The main dashboard gives a unified feeling – a user sees one system. For example, a tile could show “Strategy A: +2.5% today (ML model suggests reducing risk)” which already combines strategy performance with an ML insight. Real-time updates on this page are via WebSocket subscriptions (for things like P&L updates or new alerts). The layout should be modular, with panels that can be rearranged or configured, as different users might care about different info first.
Trading Engine Control & Monitoring (Nautilus Trader View): This section focuses on Nautilus Trader’s internals – essentially a control panel and detailed monitor for all trading activities. Here we draw inspiration from Freqtrade’s web UI, which provides rich monitoring and control over a running bot
medium.com
medium.com
. Key components of this view:
Strategy List & Controls: A table or list of all strategies (or bots) currently configured/deployed. For each strategy, display its status (running, stopped, error), current positions, accumulated profit/loss, and perhaps strategy-specific metrics (like Sharpe ratio or win rate if computed). Each strategy entry could have control buttons: Start, Stop, Pause, Resume. This allows the user to launch or halt strategies from the UI. (If only one strategy instance runs per Nautilus deployment, this might be simpler – but Nautilus likely can run multiple strategies or multiple markets concurrently, so handling multiple is beneficial.)
If a strategy is running, the user might also have controls to adjust certain parameters in real-time. For example, toggling a flag (perhaps switching the ML override on/off for that strategy), or adjusting risk limits (like max position size) if the strategy supports dynamic config changes. These could be small input fields or sliders in the UI, which when changed call an API to update the strategy’s config on the fly.
The UI should reflect state changes immediately. For instance, clicking “Stop” on a strategy should update its status indicator to “Stopped” and perhaps remove its subscriptions to market data.
Broker/Exchange Status: Since Nautilus handles broker/exchange connectivity, the UI should show the state of those connections. For example, “Connected to Binance (WS stream active)” or “Interactive Brokers – connection lost (reconnecting…)", along with account info like equity, margin availability, etc. This helps users quickly see if external dependencies are OK.
Live Market Data & Charts: A core part of a trading UI is visualizing market data and the strategy’s actions. The UI can provide a charting panel (perhaps using a library like Plotly or TradingView chart) where one can select an instrument and see price candlesticks in real-time, with overlays of technical indicators and the strategy’s buy/sell signals plotted. Freqtrade’s UI does this by allowing multi-pane charting with indicators and marked buy/sell points
medium.com
. We would integrate Nautilus’s indicator outputs here as well – since Nautilus can compute technical indicators, those time series can be plotted on the chart or below it. The user should be able to switch which instrument or strategy they are viewing, and the chart updates accordingly (with data coming via a mix of recent history from the backend and live ticks via WebSocket).
Trade and Order Monitor: A real-time table listing recent and pending orders/trades. This is like a blotter – each row might show timestamp, instrument, action (buy/sell), quantity, price, order status (filled/partial/cancelled), and maybe which strategy (if multiple). This table updates live as orders are issued or trades execute. It provides transparency into what Nautilus is doing moment-to-moment. Users might also get controls here to cancel orders or modify them if the system allows.
Logs & Runtime Info: To facilitate debugging, this view could include a scrolling log panel (tail of important logs from Nautilus). For example, log lines about risk checks, or warnings/errors. Alternatively, a link to a centralized log system (ELK stack) could be provided. But an embedded lightweight log viewer filtered to a strategy or system is useful for a quick look.
Manual Override/Control: In scenarios where the user might want to intervene, the UI could allow manual trade entries or exits on this page. For instance, a button to “Execute Market Buy” on a selected instrument (which sends an order via Nautilus immediately). Freqtrade’s UI even allows forcing trades for testing
freqtrade.io
. This should be clearly indicated (and probably permission-controlled) since it directly bypasses the strategy logic, but it’s a useful feature for emergencies or experiments.
Overall, this trading view should empower the user to see everything Nautilus is doing and to control it. The design should ensure critical data (like current positions and P&L) are very clear, and any anomalies (errors, disconnects) are highlighted. It’s effectively the replacement for a trader’s Bloomberg terminal but focused on the algorithm’s activity. All these elements are updated in real-time (via websockets or periodic refresh for things like log text). The Freqtrade UI demonstrates many of these features (trade overview lists, charts with signals, performance metrics) and proves that a web UI can effectively manage a live bot
medium.com
medium.com
. Our UI will follow a similar paradigm, tuned for Nautilus’s specific capabilities.
Machine Learning Dashboard (ML Extension View): This section is dedicated to the “ML brain” that complements the strategies. Its purpose is to surface the outputs and status of the ML models, and allow the user to manage experiments and model lifecycle. Components here include:
Current Model Status & Live Signals: At the top, show which model (or models) are currently active in influencing strategies. If the ML extension uses one primary model, display its name/version and a concise summary of its performance (e.g. “Model v3.2 – predicting 5min price direction, 68% confidence bullish”). If multiple models (for different strategies or ensemble), list each. Alongside this, present the latest predictions coming from the model in real-time. For example, a small chart or gauge updating every minute with the model’s prediction (up/down probability, or a regression output like next price). This is the live inference monitoring – akin to seeing what the “AI brain” is thinking right now. We should also show an uncertainty or confidence measure for each prediction, since knowing the confidence is important for trust (the UI might even visually emphasize when the model is unsure). Providing confidence scores or similar metrics improves transparency in human-AI decision making
medium.com
.
Model Performance Metrics: Even during live trading, the model’s recent performance can be tracked. We can show metrics like the model’s accuracy over the last N predictions, its precision/recall if it’s classification, or profit attribution (e.g. how much of the P&L can be credited to following the model’s signals). These could be shown as small dashboard widgets or charts (e.g. a line chart of rolling accuracy over the last week). If the ML is doing adaptive learning, also indicate when it last updated.
Backtest & Experiment Results: A key feature for an ML dashboard is to compare different models and configurations – essentially an experiment tracker. We will integrate a view that lists recent model training runs, backtests, or hyperparameter optimization results. Each entry could include: model version or experiment name, date/time run, parameters (e.g. “XGBoost with depth=5” or “NN with 3 layers”), and resulting performance metrics (validation accuracy, Sharpe ratio in backtest, etc.). The UI could allow selecting two or more experiments to compare their metrics side by side (for example, comparing backtest equity curves or ML validation loss curves). This is similar to how MLflow’s tracking UI lists experiment runs and lets you compare metrics across runs
artiba.org
. By visualizing these comparisons, the user can make informed decisions on which model to deploy. For instance, the UI could show that Model A had a higher backtest profit but Model B had lower drawdown and the user might choose between them. If we incorporate charts, we might show overlayed equity curves or bar charts of key metrics for selected runs, much like MLflow or other experiment trackers do when comparing runs
mlflow.org
.
Model Configuration & Deployment Controls: Users should be able to trigger actions on the ML extension from the UI. Examples: a button to “Retrain Model” using the latest data (the ML container would then start its training job, perhaps asynchronously, and the UI can show status/progress – potentially integrating with a job scheduler or just showing that a training is in progress). Another example is hyperparameter optimization (HPO) – the UI might present a form to input HPO settings or select a predefined HPO routine, and a button to run it. This would kick off an experiment (maybe orchestrated via an Airflow DAG or an internal scheduler). In an advanced setup, this could tie into an Airflow or Kubeflow Pipelines UI, but a simpler approach is to have the ML container handle the pipeline internally when triggered. We can take inspiration from Airflow’s UI which allows manual triggering of DAGs from the interface
airflow.apache.org
 – similarly, our UI’s “Retrain” button is effectively triggering a pipeline run for training. Also, if multiple models are available, the UI should allow the user to deploy/switch the active model. For example, after comparing experiments, a user might click “Activate Model v3.2 for Strategy A”, which updates the configuration so that Nautilus now uses the new model’s predictions. This could simply call an API in Nautilus or the ML service to load the new model. The UI should make it clear which model is currently live. Perhaps an indicator “(Active)” next to the model name, and others marked as “idle” or “available”.
Feature & Data Monitoring: Depending on the complexity, we might also show some of the ML model’s input features or data checks. For instance, a small table or chart of the latest feature values feeding the model, or data distribution shifts. This can help debug issues if the model behaves strangely because of out-of-distribution data. It ties into ML monitoring best practices (like checking for data drift). If using a tool like EvidentlyAI or custom stats, those could be shown here as well (e.g. “feature X mean has changed by 2σ in last hour”).
This ML view essentially brings in capabilities usually found in ML-specific dashboards (like those in MLflow, Evidently, or Neptune.ai for monitoring). We want the user to have one place to both monitor the model and trigger experiments. An example of integration is showing that a model retraining was done (perhaps last night, T+1) and its outcome – e.g. “Model v3.2 trained on Aug 7 data, validation accuracy 0.84 – now deployed”. In fact, each training or experiment can be recorded and displayed in a list (like a mini experiment tracker). This provides traceability over model versions and their performance, which is critical in a trading context to avoid regressions.
Traceability & Audit (Linking ML and Trades): A special feature of this integrated UI is the ability to trace which ML outputs influenced which trading decisions. This can be achieved by augmenting the data that Nautilus logs or streams with model references. For example, when the strategy decides to place a trade, it could include in the event metadata something like “model_signal_id” or the model’s prediction at that time. The UI can expose this in a few ways:
In the Trade/Order table of the trading view, include a column or icon indicating if a particular trade was influenced by the ML model. Hovering or clicking could reveal details: e.g. “Trade triggered by Strategy A. ML model v3.2 predicted +1.5% 5min return (confidence 70%).” This tells the user why the trade happened in terms of the ML’s input.
A Trace View or Modal: We can implement a detailed modal that appears when you click a trade or an ML prediction. For a given trade, it would show the chain of inputs: technical indicator values at the time (from Nautilus), model prediction at that time, and the final decision (buy/sell or no action). Conversely, for a given model prediction, show if it led to a trade or was ignored by the strategy logic. This is essentially an audit trail. It improves trust and debuggability; if a user asks “Why did the system buy 100 ETH now?”, they can find the trade in the UI, see that the RSI was low and the ML predicted a price increase, thus both conditions met and the strategy went long.
Timeline visualization: Another approach is a chronological timeline combining events from both systems. For example, a scrolling timeline where events are plotted: market events, strategy decisions, orders, and ML events. These could be color-coded (blue for data, green for trades, orange for ML signals, etc.). A time cursor could allow the user to pause at a specific moment and inspect what the state was (like an augmented replay). While complex, this gives a holistic view of system behavior. A simpler version could be implemented as a filterable log: entries like “09:30:00 – Price spike detected (Nautilus indicator X > threshold). 09:30:00 – Model predicts drop (70% confidence). 09:30:00 – Strategy overrides trade (sell order placed due to model override). 09:30:01 – Sell order filled at $Y.” This text timeline, possibly with expandable details, is immensely useful during post-mortems of trading sessions.
The UI should emphasize traceability to meet the requirement of knowing exactly which model influenced each trade. In regulated environments, such audit trails are important. For the developer, it’s also a debugging tool to tweak either the strategy or the model. Achieving this requires the backend to tag or log events with consistent identifiers (e.g. a unique ID for each model version and each prediction). The frontend then simply presents these linkages. Modern ML monitoring tools and experiment trackers stress lineage (which data/model led to which outcome), and we mirror that concept here in a custom way.
Configuration & Settings: Finally, a section (or modal dialogs) for system configuration and user settings is important. This could include:
Editing connection settings (like API keys for brokers or data feeds) – though these might be more static and configured via files, it’s useful to at least display them or allow a re-connect action if a key is updated.
Adjusting global system parameters – e.g. turning the entire system from live trading to paper trading mode, or a “kill switch” that immediately flat positions and stops all strategies (a big red button in the UI for safety).
Experimentation settings – for example, choosing where to store model artifacts, or toggling whether nightly retraining is automatic or manual.
UI preferences – theme (dark/light), refresh rates, etc., though these are minor.
The configuration changes that can be made live should be carefully controlled. For instance, updating a strategy’s config might require restarting it in Nautilus if not dynamically supported. The UI could enforce that by providing an “Update & Restart” button for a strategy which sends the new config and restarts the container or process.
Real-Time Data Flow and Updates
To ensure the UI remains in sync with the fast-moving system, we combine push-based updates for live data with pull-based queries for on-demand or historical data. Here’s how data flows for various needs:
WebSocket Streams (Push): Both Nautilus Trader and the ML extension will push events. The UI (via the gateway) might maintain separate channels or topics: e.g. one channel for trading events (orders, trades, errors) and one for ML events (predictions, model updates). The frontend subscribes to both. We can format messages in JSON, with a type field to distinguish event kinds. This way, components on the frontend can listen and update the relevant UI parts. For example, a new trade event immediately appends to the Trade table and updates P&L counters. A new prediction event updates the ML signal display. Using WebSockets ensures low-latency updates, which is essential for a “monitor and control” interface. It’s the same tech that trading exchanges use to feed live prices to UIs, so it’s appropriate here as well. In implementation, one can use libraries or frameworks (e.g. Socket.IO or FastAPI’s WebSocket support) to broadcast events from the backend to all connected UI clients. If needed, we might implement a small pub-sub in the gateway: the Nautilus container could POST events to the gateway which then emits over WebSocket. The ML container similarly sends events.
REST APIs (Pull): For actions initiated by the user or for loading initial state, REST endpoints are used. These include: retrieving current state (e.g. list of strategies and their statuses on page load, list of recent trades, list of experiments), and sending control commands (e.g. “start strategy X”, “trigger retraining”). The UI likely performs a REST GET on page load to get the latest snapshot, then relies on WebSocket for updates. For less frequent data (like an old backtest result), the UI can request it on demand (maybe even just fetch from a database or from the ML service when the user clicks a particular experiment). REST is suitable for these request/response interactions and is easier to implement for commands (with proper acknowledgment/response handling).
Prometheus & Metrics Query: As discussed, metrics like time-series data or aggregated stats will be pulled from Prometheus. The UI might integrate this via Grafana-like panels or custom queries. For example, to draw a chart of P&L over the last 24h, the UI could hit the Prometheus API for the metric strategy_pnl{strategy="A"} over a time range. Prometheus can return data in JSON which we then plot on the frontend. Alternatively, if we embedded Grafana, we could just show pre-configured dashboard panels (Grafana has a concept of embedded iframe panels). However, since the question expects a custom UI design, we assume direct integration without requiring the user to use Grafana separately. We can set up PromQL queries in the frontend code to get the needed metrics. This adds a bit of complexity (the UI needs to know Prometheus endpoint and query syntax), but it provides flexibility to display any metric from any container in one place. We should also configure some alerts in Prometheus (e.g. if latency > X or if model confidence is consistently low, etc.) and those alerts can feed into the UI’s alerts panel (perhaps via a simple endpoint the gateway exposes, or by the gateway subscribing to Alertmanager notifications).
Security & Access: Since this is a developer-focused UI likely running locally or on a secured network, we might not emphasize multi-user access control. However, if needed, the API gateway can handle authentication (login for UI) and role-based permissions (e.g. maybe a “viewer” role that can monitor but not execute trades). This is secondary in this design but worth noting if the system is used by multiple users or in production environments.
In summary, the UI will always have a fresh view of the system state by a combination of continuous pushes for the latest changes and the ability to query any data on demand. This hybrid approach is standard in modern monitoring dashboards – for example, a trading UI might use websockets for price updates but REST for pulling an order history. Our design adopts that for both Nautilus and ML components seamlessly.
Control Mechanisms and User Interaction
A critical part of this UI is not just viewing data but also controlling the system. We outline some of the key interactions the UI must facilitate, and how they are implemented:
Starting/Stopping Strategies: The UI provides buttons or toggles to start or stop a trading strategy (or the whole trading engine). Under the hood, this will call Nautilus’s control API – perhaps Nautilus Trader can be started with a set of strategies configured, but it might allow dynamically loading/unloading strategies via code. If direct dynamic control is not in Nautilus’s base, one approach is to have each strategy run as a separate process or container that can be started/stopped (e.g. via Docker APIs or a process manager). The UI could then interface with that manager. Given Nautilus is containerized, an alternative is that each strategy is just a config entry and Nautilus is always running; in that case, the “stop” might deactivate trading within the strategy (Nautilus could listen for a command to disable a strategy’s orders). In any case, the UI should reflect these actions immediately (optimistically updating the status to “stopping…” then “stopped” on success, etc.).
Executing Manual Trades: As mentioned, the UI might allow sending a manual trade. This involves a form where the user picks instrument, side (buy/sell), quantity, maybe order type, and submits. The backend (Nautilus API) would place the order via the broker integration. The UI then watches for the resulting order/trade event via WebSocket to update the UI (to confirm execution). Manual interventions should be logged and perhaps marked distinctly in the UI (“manual” vs “strategy” trade). This gives the user the feeling of full control when needed, similar to how some algorithmic trading UIs allow overriding or manual orders if the human sees something the bot doesn’t.
Triggering ML Tasks: On the ML side, interactions include starting an offline training job, as discussed, and possibly scheduling. If the user wants to schedule something (say daily retraining at 6 PM), the UI could either instruct the ML service to schedule it or advise the user to set it up. A simple implementation is just having a “Run now” and the knowledge that perhaps a cron in ML container will run daily anyway (with results showing up). We could integrate with an existing scheduler UI like Airflow’s, but to keep it unified, perhaps not needed unless their pipeline is complex. We can emulate parts of Airflow UI: for example, listing upcoming scheduled jobs or last run status of the “daily retrain DAG”, and a button “Run now” (which would mirror how Airflow’s UI allows manual DAG runs
airflow.apache.org
). If using Airflow under the hood, you might even embed Airflow’s UI for that DAG alone, but likely overkill. Instead, implement a simple status and trigger in our UI.
Configuring Parameters: Both strategy parameters and model hyperparameters should be adjustable. A strategy might have thresholds (e.g. an RSI level to trigger a trade) or trade size limits. Rather than editing a text config file and restarting, our UI can provide input controls (sliders, number inputs) to tweak these. When changed, an API call updates the parameter in memory (if supported) or writes to config and signals the strategy to reload it. Similarly for ML, maybe allow adjusting the threshold for model override (e.g. only override trades if model confidence > 80%). This essentially becomes part of the traceability too (recording what threshold was in effect for a given trade). For hyperparameters of model training, the UI might not allow arbitrary changes (those are usually set in code or config), but we could allow selection among presets (like choosing a different feature set or model type from predefined options before retraining).
Multi-user and Collaboration: If this UI is used by multiple team members, one might incorporate features like saving dashboard layouts or having user-specific settings. However, since the prompt doesn’t emphasize it, we consider it primarily for a single operator at a time (like the developer or quants running it).
The design of the UI should make these interactions intuitive. Buttons should be clearly labeled (e.g. “Start Strategy” or “Retrain Model”), possibly with confirmation dialogues for risky actions (“Are you sure you want to stop all trading?”). The layout could group controls with related info – e.g. on the trading view, near each strategy’s metrics, have its control buttons, so context is clear. For the ML view, next to the list of experiments, have a button “New Experiment” or “Retrain using latest data” with a small form. Also, feedback is key: when the user triggers something, they should see the effect or at least a confirmation. This could be a toast notification (“Retraining started...”) and then perhaps a status indicator (“Model training 40% complete… done”). If a training job fails, the UI should surface that error (maybe via the logs panel or an alert).
Drawing from Existing UI Design Patterns
We are effectively combining concepts from multiple domains (trading and MLOps), so it helps to lean on proven patterns from each:
Trading Dashboard Patterns: Freqtrade’s FreqUI is a great example of a unified bot interface that our design expands upon. It provides monitoring of open trades, performance metrics, and control to start/stop the bot
medium.com
medium.com
. We plan similar capabilities but extended with ML. Traditional trading platforms (think Interactive Brokers or MetaTrader interfaces) also emphasize real-time charts, order management tables, and account status – our UI incorporates those elements in the Trading view. The difference is that our UI is for a custom system, so we tailor it to developers (showing logs, internal metrics) more than a generic trading app would.
Grafana for Metrics: Grafana dashboards are highly effective at displaying time-series data from multiple sources in a unified way. We emulate Grafana by including panels for various metrics (P&L, latency, etc.), each with time-range controls and possibly the ability to zoom/scroll. Grafana’s principle of combining data sources means our UI should handle data from both Nautilus and ML in the same graphs if needed (e.g. plotting a model’s prediction probability and the market price on one chart to see correlation). We may not provide the full Grafana editing experience (no need for the user to define queries manually via UI), but predefine useful visualizations. Under the hood, since Prometheus is the source, it’s straightforward to integrate these. In short, we borrow Grafana’s real-time visualization strength and flexibility in layering data. As an example, the UI could mimic Grafana’s grid layout where each panel is movable/resizable – but that might be overkill. At least, a multi-column responsive layout with charts and tables is advisable.
Airflow for Pipeline Control: Airflow’s web UI shows DAGs (directed acyclic graphs of tasks) and their run status, and allows manual triggers and pauses. While our UI is not scheduling many pipelines, the ML training process could be seen as a pipeline (data prep -> train -> evaluate -> deploy). We might not draw a DAG graph, but we ensure the user can see the progression of these steps. For instance, if training is in progress, show sub-status like “Downloading data… training model… evaluating… done.” This gives a feel of a pipeline run similar to how Airflow shows task progress. Also, Airflow’s manual trigger and pause features are analogous to our “trigger retrain” and possibly a toggle to enable/disable automatic retrains. In Airflow UI you can turn DAGs on/off and trigger them on demand
airflow.apache.org
; in our UI, an “Auto Retrain: On/Off” switch could serve a similar purpose (with off meaning we’ll only retrain when user clicks). The concept of pausing a component can also apply to strategies (pause a strategy’s operation without fully stopping the process – maybe Nautilus can implement pausing order placement but keep market data running).
MLflow for Experiment Tracking: MLflow’s UI is a reference for how to list and compare ML experiments. It shows a table of runs with columns for metrics, hyperparameters, etc., and allows selecting runs to compare with detailed metrics charts
artiba.org
mlflow.org
. We mirror this by having an Experiments/Backtests table in the ML view. Each row might correspond to a backtest or training run. By clicking on one, you get details (maybe a new page or modal) with all logged metrics (e.g. confusion matrix, feature importance, etc., if available). By multi-selecting some runs and clicking “Compare”, the UI could generate an overlay chart (for example, cumulative return curves of two different strategies/models). This is extremely useful for a developer to assess improvements. We should also provide a way to promote an experiment to production (similar to how MLflow lets you register a model). In our UI, maybe a “Deploy this model” button on an experiment row will set that as active for the strategy (with appropriate confirmations).
EvidentlyAI / Model Monitoring UIs: Tools like Evidently provide dashboards for data drift, feature drift, and model performance over time. We can incorporate similar widgets in the ML view’s performance section – e.g., “data drift: low/medium/high” indicators, or distribution charts comparing training data vs recent data. This helps ensure our UI not only focuses on trading outcomes but also on the health of the model. A simple example: if the distribution of a key feature (say volatility) shifts drastically, a panel in the UI could flag this, warning the user that the model might need retraining or might not be reliable until retrained.
General Web UI/UX Best Practices: Since the user specifically wants a developer-ready interface, we will keep the UI clutter-free and informative. Short paragraphs of text or tooltips can explain complex metrics (so the user doesn’t have to guess what “Feature drift” means, for example). We will also ensure the UI is responsive (likely the user might run this on a desktop mostly, but if it’s web-based, responsiveness is a plus). A light vs dark theme toggle (like Freqtrade’s UI provides
freqtrade.io
) can be included given developers often prefer dark mode for dashboards.
In terms of technology, a modern React (or Vue/Angular) frontend would be suitable, using component libraries for charts (e.g. D3 or Plotly for custom charts, or Recharts). For the layout, a grid system helps to arrange the panels logically. We should also consider performance: lots of real-time data means we should efficiently update components (possibly using a state management library; for example, one case study described using React + Recoil to manage incoming WebSocket data smoothly
medium.com
). Ensuring that the UI can handle bursts of data (like many trades in a short time) without freezing is important – using web workers or debouncing updates for charts might be needed.
Conclusion
The proposed frontend architecture provides a holistic dashboard for a containerized Nautilus Trader + ML system, enabling developers to monitor and control every aspect in one interface. By drawing on patterns from trading dashboards (for live market and execution data) and ML platforms (for experiment tracking and model monitoring), we ensure that the UI addresses both sides of the system equally. The component layout separates concerns (trading vs ML) but keeps them closely accessible, reinforcing that this is a single integrated platform. This design emphasizes real-time insight (through WebSockets and live charts) and actionability (through controls to adjust strategies or retrain models). It also bakes in transparency – from performance metrics to decision traceability, a user can dig into what’s happening under the hood. For example, if an ML-driven trade goes wrong, the user can trace back and see the model’s role in that decision immediately, then perhaps tweak the model or strategy accordingly. By implementing the above architecture, developers and quants using the system will have a powerful “command center” to iterate faster. They can observe how their ML models and strategies behave in concert, compare alternative configurations quickly, and maintain confidence that they can intervene or debug as needed. This unified UI turns the containerized Nautilus+ML deployment from a black-box engine into an interactive, transparent, and tunable trading+AI platform – much like combining a trading cockpit with an ML control panel. It exemplifies how modern tooling (like Prometheus for metrics and web dashboards) can be leveraged to create an environment where both algorithmic trading and machine learning elements are first-class citizens, fully visible and steerable by the user.
