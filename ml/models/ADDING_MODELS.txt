Great question. The short version: make “adding a new model family” boring by standardizing the contract (features in, probabilities out), exporting to ONNX wherever possible, and falling back to teacher→student distillation when it’s not. Below is a concrete, plug‑and‑play plan you can implement now, plus a compatibility matrix for the libraries you listed.

The contract (don’t skip this)
Every model you train—no matter the library—must produce the same serving contract so your MLSignalActor doesn’t care where it came from:

Inputs (feature schema): ordered names, dtype=float32, length N, optional scaler hash.

Outputs (output schema): one of

binary_proba → shape [batch, 1], value = p(up)

regression → shape [batch, 1], value = real scalar (if you will threshold it later, add a calibrator)

multihorizon_proba → shape [batch, H] (e.g., 1s/5s/30s horizons)

multiclass_proba → shape [batch, K]

Sidecar metadata (JSON): feature_names, feature_dtypes, schema_hash, output_schema, best_iteration (if trees), calibration (Platt/isotonic params), trainer_version, train_date_range, instrument(s), flags (what indicators/L2 were used).

Your loader must hard‑fail if the live feature vector (names/order/dtypes/hash) doesn’t match the sidecar. Your actor should never guess.

The plugin shape (so new families drop in)
Create two thin interfaces and one acceptance test:

1) TrainerAdapter (per family)
python
Copy
class TrainerAdapter(Protocol):
    def fit(self, X_train, y_train, X_val, y_val, cfg) -> Any: ...
    def to_onnx(self, fitted_model, example_input, cfg) -> Path: ...
    def save_metadata(self, fitted_model, onnx_path, feature_schema, output_schema, cfg) -> Path: ...
to_onnx must return a single ONNX file containing either just the model or model+pre/post (see below).

If the family cannot export to ONNX, this adapter should train a student (e.g., LightGBM / small PyTorch MLP) on the teacher’s outputs and export that to ONNX.

2) InferenceAdapter (optional)
You already have ONNX in the actor, so most families won’t need a runtime adapter. Only use one if you must route multiple outputs or chain graphs. Otherwise rely on onnxruntime.

3) Acceptance test (per family)
Run framework-native predict on a validation slice.

Run onnxruntime on the same inputs.

Assert max|diff| ≤ tolerance; save report; refuse registration if it fails.

Pre/post: where to put scalers & calibration?
Pick one and stick to it:

Embedded graph (preferred): include standardization (and even a sigmoid/softmax or Platt) inside the ONNX graph. Then your actor gets p(up) directly.

Sidecar transforms: if embedding is painful, persist scaler/calibrator in metadata and apply in the actor. If you do this, keep it tiny (no pandas), pure‑NumPy float32.

PyTorch (plain, pytorch‑forecasting, neuralforecast)

PyTorch Documentation
PyTorch

Gotchas: set dynamic axes for batch; freeze any non‑Tensor inputs (categorical vocab maps, horizons, etc.) into constants or pre/post in the graph.

pytorch‑forecasting / neuralforecast: both are PyTorch‑based—export is usually fine if you call the underlying nn.Module with the processed tensors you’d feed during serve. (Avoid exporting the high‑level “trainer”; export the core model.)

PyTorch Geometric (PyG)
Export: No upstream ONNX support right now; maintainers recommend TorchScript. 
GitHub

Plan: Use PyG models as teachers offline → train an ONNX student (trees or small MLP) on their outputs. If you need direct serving, run a tiny TorchScript microservice and keep the bus contract identical, but your actor won’t load it natively.

statsmodels / arch (ARIMA/GARCH families)
Export: No official ONNX converters for statsmodels; same story for arch (GARCH). 
GitHub
Stack Overflow

Plan: Treat them as teachers (risk/vol/seasonality nowcasts). Distill to an ONNX student (e.g., linear or tree) or compute their numeric outputs offline and train a student to emulate them from L1 features.

River (online/streaming)
Export: River is designed for in‑process updates, not ONNX export. Use it for online adapters (e.g., a tiny logistic that slowly re‑calibrates) in a separate actor, or distill its behavior to an ONNX student periodically. 
riverml.xyz

EconML / CausalML (causal estimators)
Export: These meta‑learners compose many models; not ONNX‑friendly by default.

Plan: Use them as slow cadence teachers to produce a risk multiplier or treatment effect you fold into regime gating (prior), then distill that mapping into an ONNX student if you need it inside the fast path.

ONNX Runtime tips (so it flies)
Providers: CPU EP by default; enable CUDA only if your graph justifies it. ONNX Runtime handles provider partitioning; CPU fallback keeps it robust. 
ONNX Runtime

Opset: pick one (e.g., 17) and freeze it fleet‑wide; newer is fine but keep it uniform.

Dynamic axes: set {"input": {0: "batch"}} so you can score mini‑batches in backtests without re‑export.

Quantization (later): post‑training INT8 can double throughput; add an acceptance test to guard accuracy.

Two patterns you’ll use again and again
Teacher→Student distillation
If it can’t export cleanly, don’t fight the toolchain. Train a teacher in the fancy library, generate labels on your training set, then fit a small ONNX‑friendly student (LightGBM or MLP) on exactly the features you have live. Calibrate, export, done. This keeps the signal actor simple and fast.

Prior × likelihood
Use slow/complex families (EconML, statsmodels/arch) to compute low-frequency priors (risk multiplier, vol regime) off the critical path. Publish that prior as a separate signal; your execution strategy multiplies it with the fast ONNX student scores. It matches how you trade: macro for bias, price for trigger.
