# Unified Feature Registry

## Overview

The Feature Registry provides a self-describing, versioned catalog for feature sets used by models. It mirrors the Model Registry’s rigor with manifests, lifecycle stages, parity/latency gating, and schema validation at serve time.

Key goals:

- Deterministic, ordered feature schemas (names + dtypes)
- Strict batch/online parity tracked via digests
- Latency/footprint constraints for hot path
- Lineage across teacher → student feature sets
- Safe promotion, deprecation, and scrapping

## Manifest

`FeatureManifest` contains:

- Identity: `feature_set_id`, `name`, `version`
- Role: `FeatureRole` (TEACHER, STUDENT, INFERENCE_SUPPORT)
- Data Requirements: `DataRequirements` (L1_ONLY|L1_L2|L1_L2_L3|...)
- Schema: ordered `feature_names`, `feature_dtypes`, `schema_hash`
- Pipeline: `pipeline_signature`, `pipeline_version`, `capability_flags`
- Constraints: `constraints` (e.g., `max_latency_ms_hot_path`)
- Quality: `parity_tolerance`, `parity_digest`, `perf_digest`
- Lineage: `parent_feature_set_id`
- Lifecycle: `stage` (CANDIDATE|STAGING|PROD|DEPRECATED|SCRAPPED)

Schema hash is computed deterministically from `(names, dtypes, pipeline_signature)`.

## Local Registry

Use `LocalFeatureRegistry` for file-based storage:

```python
from pathlib import Path
from ml.registry.base import DataRequirements
from ml.registry.feature_registry import (
    FeatureManifest, FeatureRole, LocalFeatureRegistry, compute_schema_hash
)

reg = LocalFeatureRegistry(Path("ml/features"))
manifest = FeatureManifest(...)
fid = reg.register_feature_set(manifest)
reg.promote(fid, FeatureStage.PROD)
```

## Generating Manifests from FeatureEngineer

```python
from ml.features.engineering import FeatureConfig, FeatureEngineer
from ml.registry.base import DataRequirements
from ml.registry.feature_registry import FeatureRole

eng = FeatureEngineer(FeatureConfig())
manifest = eng.generate_feature_manifest(
    name="default",
    version="1.0.0",
    role=FeatureRole.STUDENT,
    data_requirements=DataRequirements.L1_ONLY,
)
```

## Actor Validation

Actors can validate schemas at startup:

```python
from ml.actors.signal import MLSignalActorConfig

cfg = MLSignalActorConfig(
    model_path="/path/model.onnx",
    model_id="my_model",
    bar_type=bar_type,
    instrument_id=instrument_id,
    feature_config=FeatureConfig(),
    feature_set_id="feature_set_...",
    registry_path="ml/features",
    use_registry_features=True,
)
actor = MLSignalActor(cfg)   # Validates schema vs manifest
```

## Training Integration

```python
from pathlib import Path
from ml.training.feature_export import register_feature_set_from_engineer
from ml.registry.base import DataRequirements
from ml.registry.feature_registry import FeatureRole

fid = register_feature_set_from_engineer(
    registry_path=Path("ml/features"),
    name="default",
    version="1.0.0",
    role=FeatureRole.STUDENT,
    data_requirements=DataRequirements.L1_ONLY,
    parity_report={"tolerance": 1e-10, "max_difference": 3e-12},
    perf_report={"p99_feature_ms": 0.4},
)
```

## Lifecycle

- `register_feature_set()` → `promote()` to STAGING/PROD after gates pass
- `deprecate()` when superseded
- `scrap()` to mark as scrapped (retains lineage for audit)

## Tests & Quality Gates

- Hypothesis-based tests verify schema hash determinism and gating.
- Parity/latency digests should be generated by validation tools and stored in the manifest.
