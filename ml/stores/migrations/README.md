# ML Stores Migrations

This directory houses every SQL change applied to the ML persistence layer—
registry tables, feature/model stores, and now the dataset-ready macro/events
stores that underpin Full Dataset Readiness.  Filenames use a **monotonic,
zero-padded prefix** (`001_…`, `015_…`, etc.) so migrations always execute in a
deterministic order and reviewers can reason about scope at a glance.

## Quick reference

| Prefix | Purpose | Status |
| --- | --- | --- |
| `001_bootstrap_schema.sql` | Canonical schema for ML stores + registry tables. | Applied by default (`_BASE_MIGRATIONS`). |
| `002`–`014` | Legacy/optional SQL kept for reference (schema hardening, views, BRIN indexes, registry patches). | Invoke manually if needed. |
| `015_macro_release_calendar.sql` | Macro ALFRED release calendar store. | Active. |
| `016_macro_observations.sql` | Long-format macro observations. | Active. |
| `017_events_calendar.sql` | Structured calendar/events store. | Active. |
| `018_microstructure_minute.sql` | Microstructure (L1) feature store. | Active. |
| `019_l2_minute.sql` | L2 depth feature store. | Active. |

### Inventory command

Use the monotonic naming scheme + checksums generated by:

```bash
python -m ml.tools.print_migration_inventory
# Example output:
# 001::001_bootstrap_schema.sql::…
# 015::015_macro_release_calendar.sql::…
```

Pass `--skip-archive` to omit historical files or `--base <dir>` for custom
paths.  Include the inventory diff when reviewing migration PRs to make drift
obvious.

## Applying migrations

- **CLI**: `python -m ml.cli.apply_migrations --schema both` runs the default plan
  (`_BASE_MIGRATIONS` → registry migrations + `001_bootstrap_schema.sql`). Add
  `--full` to layer on optional files in lexical order.
- **Integration manager**: `MLIntegrationManager(auto_migrate=True)` uses the
  same planner.  Control scope via `ML_MIGRATIONS_SCHEMA=stores|registry|both`
  and include optionals with `ML_MIGRATIONS_FULL=1`.
- **Migration runner**: `poetry run python -m ml.stores.migrations_runner apply --db-url ...`
  executes every file in lexical order (default path `ml/stores/migrations`).

All scripts are idempotent and safe to re-run.  The `PartitionManager` performs
post-migration maintenance to keep partitions in sync, and DDL-heavy tests must
remain `@pytest.mark.serial`.

## Canonical process for new migrations

1. **Choose the next prefix** (three digits, strictly increasing). Avoid
   recycling numbers—future diffs rely on monotonic ordering.
2. **Document intent** at the top of the SQL file using the standard headers:
   ```
   -- Migration: <short summary>.
   -- Rollback: <manual steps or DROP statements>.
   ```
3. **Update this README** with the new prefix/description.
4. **Record the inventory** by running `python -m ml.tools.print_migration_inventory`
   and paste the output (or diff) into the PR/issue for auditing.
5. **Validate on a clean Postgres instance** via
   `poetry run python -m ml.stores.migrations_runner apply --db-url <test-dsn>`
   to catch ordering or checksum issues early.

Following this checklist keeps migrations deterministic, auditable, and aligned
with the Full Dataset Readiness guardrails.
