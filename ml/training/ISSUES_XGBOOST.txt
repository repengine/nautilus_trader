XGBoostModel (inference wrapper) — observations & fixes
File: xgboost_model.py

What’s good

One wrapper that accepts both a raw Booster and sklearn XGBClassifier/Regressor. Simple, production-friendly API that returns np.ndarray[float32].

Converts 1D → 2D, handles predict_proba for sklearn classifiers. Clean.

Issues / improvements

Booster vs sklearn detection can misfire.
You infer “Booster” by the absence of predict_proba. An XGBRegressor has no predict_proba but is not a raw Booster. Prefer type checks:

isinstance(model, xgb.Booster) → Booster branch

isinstance(model, (xgb.XGBClassifier, xgb.XGBRegressor)) → sklearn branch
This avoids calling Booster-style predict on sklearn objects (undefined/slow behavior).

DMatrix creation per tick is expensive. Use inplace_predict.
For raw Booster, replace:


dmatrix = xgb.DMatrix(features)
preds = booster.predict(dmatrix)
with:


preds = booster.inplace_predict(features, validate_features=False)
It avoids DMatrix allocation and is meaningfully faster for single-row, high-frequency inference.

Honor early-stopping best iteration.
Your trainer captures best_iteration; the wrapper should use it when predicting a Booster, e.g. iteration_range=(0, best_iteration+1). Read it from the metadata you already pass into the wrapper (see trainer’s training_metadata). If unavailable, fall back to full trees.

Shape/semantics normalization.
Guarantee consistent outputs:

Binary cls → 1D positive class probability (float32).

Multiclass → 2D matrix of class probs.

Regression → 1D float.
Right now multiclass via sklearn will be 2D, but Booster path returns 1D for binary and may return margins depending on objective—be explicit.

Input validation beyond shape.
Add (cheap) checks against metadata: number/order of features and dtype. Hard-fail if mismatch to prevent silent garbage inputs.

XGBoostTrainer (training/export) — observations & fixes
File: xgboost.py

What’s good

Clear prepare_data() with feature list & metadata; optional NaN handling via config.

GPU toggle, monotone constraints, early stopping, eval logs, Optuna hooks—nice.

Saves native XGBoost JSON (not pickle) + writes a sidecar metadata with feature names, metrics, and config.

ONNX export path intentionally normalizes feature names to f0..fn to avoid converter quirks—good.

Issues / improvements

Trainer predict() returns hard labels for binary objectives.
That’s great for evaluation by accuracy, not for trading. Keep trainer predict() returning probabilities; do thresholding in the strategy. Provide an optional return_proba=True or a separate predict_proba.

Use best iteration at predict time.
In both training-time eval and exported inference, apply iteration_range=(0, best_iteration+1) (or equivalent). Otherwise, predictions may use all trees beyond the early-stopped point.

GPU settings version-proofing.
You set tree_method="hist" and device="cuda:n". That’s XGBoost 2.x style. For 1.x compatibility, consider:

If v1.x → tree_method="gpu_hist" and predictor="gpu_predictor"

If v2.x → your current device param is fine
Add a small version guard so GPU actually engages on all environments.

NaN handling policy.
You sometimes fill NaNs (handle_missing → fill_nan(value)). XGBoost natively supports NaN; for microstructure, NaNs often “mean something” (gaps). Prefer leaving NaN and pass missing=np.nan to DMatrix. If you must fill, store the fill value in metadata and replicate it in live features.

Metadata completeness for serve-time safety.
You already persist feature_names, metrics, and config. Add:

best_iteration, objective, eval_metric

feature dtypes, order, and a schema hash

class mapping (for multiclass)

training date range, instrument(s)
Your loader/XGBoostModel can then refuse to score on mismatch.

Output shape in metadata.
You set "output_shape": [None, 1] always. That’s wrong for multiclass. Make it dynamic based on objective / classes.

ONNX export notes.
You already reset names to f0.. and set opset 12. Also decide whether scalers/calibration live inside the ONNX graph (preferred for parity) or as sidecar transforms—document and persist.

Calibration & imbalance.
You support scale_pos_weight. Add optional calibration (isotonic/Platt) and save its artifact + config in metadata. Live strategies depend on calibrated probabilities.

Consistent dtype across pipeline.
Training uses float64, wrapper returns float32. Choose one for the whole pipe (I’d go float32) and enforce it in prepare_data() and in the wrapper’s validator to avoid silent cast costs.

Feature importance & SHAP.
All good for analysis. Just ensure these utilities never run in hot paths. (They don’t today.)

Integration insights (how these two fit NautilusTrader)
Serve-time safety: use your sidecar metadata from the trainer to validate the live feature vector before every score (names, count, dtype, hash). Fail fast. (You already write training_metadata.feature_names—extend it and wire the check in the wrapper/loader.)

Latency: in the Nautilus actor, pre-allocate a np.float32 row buffer for features and call Booster.inplace_predict (or ONNXRuntime later). Avoid Pandas/Polars in hot loop.

Thresholding: keep thresholds and cooldowns in the strategy (not the trainer). You’ll want to A/B and regime-gate these later.

Best-iteration propagation: trainer stores it → metadata → wrapper reads it → predict with iteration_range. This keeps backtest/live parity with early stopping.

Quick patch list (actionable)
Type detection fix in XGBoostModel.__init__ (use isinstance checks).

Swap to inplace_predict in Booster path; add best-iteration handling via metadata.


Trainer predict() returns probs (add return_proba flag); never hard-threshold by default.

Add version guard for GPU params; set predictor="gpu_predictor" on v1.x.

Metadata++: include best_iteration, dtypes, schema hash, training window, output shape. Enforce in loader/wrapper.

Unify dtype to float32 end-to-end and ensure the wrapper’s validate_input checks it.

Calibration hooks: add optional isotonic/Platt post-fit + persist; strategy uses calibrated probs.
